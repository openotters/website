`ottersd` is the long-running daemon process that hosts your agents. It
owns the agent loop, the local image store, persistent state (sessions,
chat history, captured tool output), provider credentials, and the
spawn environment that keeps host secrets out of the agent's reach.

## Local layout

By default, `ottersd` keeps state under `~/.otters/`:

```
~/.otters/
├── agents/          # per-agent runtime state (workspaces, agent.yaml)
├── logs/            # per-agent runtime logs (one file per agent UUID)
├── providers.yaml   # registered LLM providers (otters provider add)
├── daemon.db        # SQLite state store (agents, sessions, jobs, mounts)
└── registry/        # embedded OCI store (system executor only)
```

The Docker executor uses Docker's image store instead of `registry/`;
that directory only exists under the system backend.

## Configuration

`ottersd serve` reads YAML on start-up and merges with CLI flags and
environment variables. Precedence (highest first):

1. CLI flag (`--executor docker`)
2. Environment variable (`OTTERSD_EXECUTOR=docker`)
3. YAML config file
4. Built-in default

### Config file locations

First match wins:

1. `--config` / `-c` flag (any path you supply)
2. `~/.otters/ottersd.yaml` — user-scoped, same directory as the rest
   of the daemon's state.
3. `/etc/otters/ottersd.yaml` — host-scoped, for systemd-managed
   daemons.

Every `serve` flag has a matching YAML key — kebab-case becomes nested
keys under `serve:`. Run `ottersd serve --help` for the full list.

### Example: `~/.otters/ottersd.yaml`

```yaml
serve:
  # Backend: 'system' (default) or 'docker'.
  # See /docs/executors for the comparison.
  executor: docker

  # Pin the unix socket / TCP listener / data paths.
  socket-path: /var/run/ottersd.sock
  http-addr: 127.0.0.1:5500
  no-ui: false

  # Pool tuning — defaults shown for context.
  max-concurrent: 10
  backoff-base: 1s
  backoff-cap: 30s
  shutdown-timeout: 5s
```

The same settings via env vars (e.g. for a systemd unit or a Colima
post-start hook):

```sh
export OTTERSD_EXECUTOR=docker
export OTTERSD_HTTP_ADDR=127.0.0.1:5500
ottersd serve
```

Or one-shot at the CLI:

```sh
ottersd serve --executor docker
```

`otters info` reports the active backend so you can confirm which one is
in effect after a restart.

## Spawn environment

Every agent runs with a locked-down env so host secrets stay out of
reach, even when the agent spawns subprocesses via `sh`. The Agentfile
can opt in extra OS env vars via the [`ENV`](/docs/agentfile#env)
instruction; reserved keys are rejected at build time:

- `PATH` — rooted at the agent's `bin/` (system) or
  `/opt/bins/<name>` (docker).
- `HOME`, `XDG_*`, `TMPDIR` — rooted under the agent's directory.
- `LANG`, `OTTERS_AGENT_ROOT` — set by the runtime.
- `*_API_KEY`, `*_API_BASE` — travel through the provider-credential
  channel; declare a [provider](/docs/cli) instead.

The lock-down rules apply to both [executor backends](/docs/executors).
